# 问题

## 场景

一个重度使用 Claude Code 的人，日常工作经常是**同时开 4-5 个
任务**。为了让这些任务并行推进，ta 会开 4-5 个终端窗口，每个
窗口里跑一个独立的 Claude Code 会话，每个会话负责一件事：

- 一个在重构登录模块
- 一个在修支付回调
- 一个在跑测试
- 一个在整理文档
- 一个在做性能压测

看起来很高效 —— 直到 ta 发现自己大部分时间都花在**切换窗口**上。

## 痛点

### 1. 注意力切换的成本

每次从一个终端切到另一个终端，大脑都要做一次**上下文重建**：

- 这个会话刚才在干什么来着？
- Claude 上一条回复说到哪了？
- 我给它的指令是什么？
- 它问过我什么还没回答？

哪怕只是 10 秒钟，5 个会话一天切换几十次，累计下来的认知负担
非常可观。更糟的是，切换本身打断心流 —— 你正在深度思考 A 的
架构问题，被 C 的一个通知拉走，等处理完 C 再回来，A 的思路已经
散了。

### 2. 并行不等于高效

开 5 个窗口看起来是 5 倍产出，但人脑的注意力是单线程的。真实情况
更接近：

> 5 个会话各跑着，但每一刻只有一个在你脑子里是"热的"，
> 其它 4 个在后台干到一半就卡住等你，你却不知道它们卡了。

结果是：**并行的是 Claude，不是你**。你依然是瓶颈，而且是一个
在高频切换中质量不断下降的瓶颈。

### 3. 重要的事容易被埋没

5 个终端同时在滚动输出，你很难一眼看出哪个最需要你关注。Claude
刚才写的那句"我觉得这里的方案可能有问题，你看一下"，滚过去就没了。
等你切回来，已经找不到在哪条消息里。

你被迫**轮询**每个终端"看看有没有需要我处理的"，这本身就是最累的
工作模式。

### 4. 切换让人不想开更多任务

因为切换太痛苦，你会下意识地限制自己并行的任务数。本来可以同时
跑 8 个的地方，你只敢跑 3 个，因为再多就顾不过来了。**工具的
心智成本限制了你的吞吐量**。

## 谁有这个问题

本项目的目标用户有这些特征：

1. **已经是 Claude Code 的重度用户**。对基本概念和工作流熟悉
2. **会同时做多件事**。做一件事的人不需要这个工具
3. **技术背景**。能接受命令行、能理解"后台进程"这类概念
4. **珍惜注意力**。认为"少切几次窗口"值得投入时间搭工具

非目标用户：

- 只偶尔用 Claude Code 的人 —— 原生体验已经够好
- 只做单一任务的人 —— 本来就不存在切换问题
- 完全不想碰命令行的人 —— 我们的第一版不优化这部分（但最终形态
  可以通过 Web 界面覆盖这部分用户，见 `design.md`）

## 现有方案为什么不够好

用户可能会想到几种"土办法"，每一种都有关键缺陷：

### 土办法 1：Claude Code subagent + 后台任务

用 Agent 工具派发并行子代理，让一个主 Claude 统筹。

- ✅ 心智上确实"批奏折"了
- ❌ **所有子代理共享一个 context window**。5 个长任务的报告
      都堆在同一条主对话里，token 消耗成倍、互相污染、跨任务
      可能混淆
- ❌ 主 Claude 只看到子代理的**最终总结**，不是过程。遇到要
      深入某一个任务讨论时几乎没法再接回去
- ❌ 子代理的生命周期绑在一次主对话里，不是真正独立的会话

### 土办法 2：tmux / zellij / 多终端窗口

用终端复用器把 5 个窗口摆在一个屏幕上。

- ✅ 视觉上统一了
- ❌ **注意力切换没有少**。眼睛从左边看到右边、从上到下扫，
      大脑依然在做 5 套上下文切换
- ❌ 依然需要你主动"巡逻"每个窗口检查状态

### 土办法 3：hooks + 事件日志

每个 Claude Code 会话用 Stop/Notification hook 往一个共享文件
写事件，你 `tail -f` 一个总览日志。

- ✅ 被动获知"有事发生"，不用主动巡查
- ❌ **只是通知，不是汇总**。你看到的还是一堆原始事件，哪条重要
      哪条不重要得你自己判断
- ❌ 反向通道（你想给某个会话发消息）没有解决，只能切回该会话手打
- ❌ 日志格式要自己定、事件要自己解析，搭建和维护成本高

### 土办法 4：5 个独立会话 + Slack 通知

每个会话完成时往 Slack 发消息。

- ✅ 手机也能看
- ❌ 没解决"谁发的更重要"的问题
- ❌ 没解决反向通道
- ❌ 你依然在 5 个会话之间切换

## 共同缺陷：**缺一个理解者**

上面所有方案都有一个共同的根本问题：**它们都只在做机械的聚合，
没有任何一层在帮你「理解」发生了什么**。

你依然要：

- 自己判断每个事件的轻重
- 自己决定先处理哪个
- 自己把原始日志翻译成"这件事说人话是什么意思"
- 自己在 5 套上下文中保持心智模型

聚合 ≠ 理解。**我们要的不是更好的仪表盘，是一个能读懂仪表盘并
用人话告诉你的人**。这就是本项目存在的理由。
